# Step 2 — Understand the Current Email Flow

## Overview

This step analyzes how the current email sending process works in MZinga. The goal is to understand the existing synchronous implementation before refactoring it into an asynchronous, decoupled architecture using a microservice.

---

## 2.1 The `afterChange` Hook

File: `src/collections/Communications.ts`

The `afterChange` hook is executed every time a `Communications` document is created or updated.

### Key Characteristic

* The hook runs **synchronously**.
* The HTTP request does **not complete** until all logic inside the hook finishes.

This creates a **blocking behavior**.

---

### Sequence of Operations

#### 1. Resolve Upload Attachments

The system iterates through the rich-text body and processes nodes of type `upload`.

* Fetches full document using `payload.findByID`
* Merges full data into the node

**Purpose:** Ensure attachments have complete data (e.g., URLs) for rendering.

---

#### 2. Serialize Rich-Text to HTML

```ts
const html = TextUtils.Serialize(body || "");
```

* Converts Slate AST into HTML
* Produces the final email body

**Note:** `TextUtils.Serialize` is independent from Payload and reusable externally.

---

#### 3. Resolve `tos` to Email Addresses

* Extracts user IDs from `tos`
* Queries the `users` collection
* Maps results to email addresses

**Important:**

* `tos` stores relationships, not emails
* Throws error if no valid emails are found

---

#### 4. Resolve `ccs` and `bccs`

* Same logic as `tos`
* Produces comma-separated email strings

---

#### 5. Build and Send Emails

* Creates one message per recipient
* Calls `MailUtils.sendMail(...)`
* Stores promises in an array

```ts
await Promise.all(promises);
```

**Critical Point:**

* Waits for all SMTP calls to complete
* This is the main bottleneck

---

### Conclusion (2.1)

The hook performs:

* Data enrichment
* Data transformation
* Recipient resolution
* Email sending

All within the same synchronous flow → **blocking and not scalable**.

---

## 2.2 The MailUtils Flag

File: `src/utils/MailUtils.ts`

### `DEBUG_EMAIL_SEND`

* Reads an environment variable
* If set to `1`:

  * Does NOT send emails
  * Logs the message instead

### Purpose

* Debugging without sending real emails

### Key Insight

This pattern can be extended to introduce a new flag such as:

```
DISABLE_EMAIL_SEND=1
```

This allows disabling email sending entirely during migration.

## 2.3 MongoDB Document Structure

Command:

```bash
mongosh "mongodb://admin:admin@localhost:27017/mzinga?authSource=admin&directConnection=true"
use mzinga
db.communications.findOne()
```

### Example Document

```json
{
  "_id": ObjectId("..."),
  "subject": "Test subject",
  "body": [ /* Slate AST nodes */ ],
  "tos": [
    { "relationTo": "users", "value": ObjectId("...") }
  ],
  "ccs": null,
  "bccs": null,
  "sendToAll": false,
  "createdAt": ISODate("..."),
  "updatedAt": ISODate("...")
}
```

---

### Key Observations

#### 1. Relationships, Not Emails

* `tos`, `ccs`, `bccs` store references:

```
{ relationTo: "users", value: ObjectId }
```

* Emails must be resolved by querying the `users` collection

---

#### 2. Body is Slate AST

* Stored as structured nodes
* Not directly usable in emails

→ Must be converted to HTML

---

#### 3. No Status Field

* No tracking of email state
* No indication of success/failure

→ Will be added in Step 3

---

### Inspect Users Collection

```js
db.users.findOne({}, { email: 1 })
```

Confirms where email addresses are stored.

---

## Final Understanding

The current system:

1. Saves a communication
2. Immediately processes and sends emails
3. Waits for all SMTP operations to complete

### Problems

* Blocking HTTP requests
* Poor scalability
* Tight coupling between persistence and delivery

### Key Insight

All necessary data is already stored in MongoDB.

This enables a new architecture where:

* The app only saves the document and emits an event
* A microservice handles email processing asynchronously

---

## Summary

* `afterChange` handles the full email pipeline synchronously
* `MailUtils` already supports environment-based behavior
* MongoDB contains enough data to decouple the process

➡️ Ready for Step 3: introducing status tracking and decoupling
